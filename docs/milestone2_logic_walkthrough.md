# Milestone 2 streaming Sobel walkthrough

## Stream contract

`conv_pipeline_top.sv` accepts raster-ordered 8-bit grayscale pixels with explicit `in_valid`, `in_x`, and `in_y`. A pixel is accepted only on a clock with `in_valid=1`; blanking gaps do not move the line buffers, horizontal shifts, coordinates, or counters. The source cannot be backpressured, and the datapath accepts one valid pixel per clock.

The border policy is cropped interior. A complete `WIDTH x HEIGHT` input creates `(WIDTH-2) x (HEIGHT-2)` outputs. The first output is `(1,1)` and the last is `(WIDTH-2,HEIGHT-2)`.

## Datapath

1. `line_buffer_3x3.sv` alternates even and odd input rows between two inferred block RAMs. On each accepted pixel it reads the old value before replacing it, making the current, previous, and two-back rows available without clearing the memories on reset.
2. `window_3x3.sv` maintains two horizontal delays for each of the three rows. It emits `p00` at the top-left through `p22` at the bottom-right only after two rows and columns exist.
3. `sobel3x3.sv` uses four stages: positive/negative partial sums, signed subtraction, absolute-value addition, and 8-bit saturation. Coordinates and valid are registered through the same stages.
4. `stream_checksum.sv` computes the standard reflected CRC-32 over valid output bytes. The checksum and per-frame output count restart on the first cropped output, which also handles consecutive input frames correctly while the pipeline is still draining the prior frame.

The optional `grayscale_rgb565.sv` block implements:

```text
gray = (77*R8 + 150*G8 + 29*B8 + 128) >> 8
```

It is tested separately so camera color conversion is ready for Milestone 3 without coupling it to the grayscale Sobel core.

## Board demo

`arty_m2_sobel_top.sv` preserves the Milestone 1 reset, debounce, heartbeat, UART, and pin mappings. A debounced `BTN1` pulse starts the pattern selected by `SW2:SW0`. `synthetic_pixel_source.sv` supplies a 320x240 frame at one pixel per 100 MHz clock. The result count and CRC are compared with values generated by `golden_sobel.py`, then `m2_uart_reporter.sv` sends a hexadecimal PASS/FAIL line.

UART remains 115200 by default, but `UART_BAUD` is a top-level parameter. The convolution always runs at full fabric speed; UART only reports the short result line after processing. A later build can use a faster supported rate such as 921600 without changing the image pipeline.

## Implementation results

- Vivado 2026.1 synthesis: 0 errors, 0 critical warnings.
- Route: 0 failed or unrouted nets.
- 100 MHz timing: WNS 2.284 ns, WHS 0.093 ns; all timing constraints met.
- Utilization: 460 LUTs, 717 registers, 2 RAMB18E1 blocks, and 1 DSP48E1.
- The two 320x8 line banks infer as block RAM. Distributed RAM usage is zero.
- The single DSP belongs to the coordinate-hash synthetic test pattern, not to the Sobel constant-by-two arithmetic.

The expected warnings are:

- Sobel center pixel `p11` has a zero kernel coefficient.
- Reserved `BTN2`, `BTN3`, and `SW3` have no logic load.
- The BRAM output registers are not absorbed into the RAM primitives; final timing still passes by 2.284 ns.
- The BRAM controls ultimately depend on the asynchronously asserted board reset. Output valid is suppressed during reset, and two fresh rows are required before any post-reset window is accepted, so old or reset-time read data cannot become a valid output.
- Buttons, switches, LEDs, reset, and UART have no synchronous external input/output delays because they are human-driven or asynchronous serial/debug signals rather than source-synchronous interfaces.
- The synthetic coordinate hash uses an unpipelined DSP, but it is outside the reusable convolution datapath and passes 100 MHz timing.
- The board configuration-voltage property is not specified in the inherited XDC. All used I/O pins are explicitly `LVCMOS33`; this warning can be removed later by recording the board configuration-bank voltage property.
